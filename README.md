# Iron Forge

**An AI-powered revenue-recovery system for service businesses.** When a roofer misses a
call, a dental office has an unpaid invoice, or a contractor's quote goes cold, Iron Forge
re-engages that customer automatically over SMS — a compliant, on-brand conversation
generated by Claude, with payments and scheduling handled inline — and recovers revenue the
business would otherwise have lost.

> Service businesses miss or fail to follow up on a large share of inbound demand. Every
> missed call and every dead quote is money already spent on marketing, walking out the door.
> Iron Forge is the always-on follow-up layer that catches it.

Built by **Ironline Response LLC**. This repository is a clean, self-hostable reference
implementation of the production stack — orchestration, database, compliance logic, prompts,
and the conversation engine.

---

## How it works

One architecture serves every vertical. Only three things change between a roofer and a
dental office: the **trigger source**, the **system prompt**, and the **objection library**.
Every trigger runs the same seven-stage pipeline:

```
  CRM / PMS / missed-call webhook
            │
            ▼
  1. Trigger ingestion        validate the event              (code_nodes/validate_payload.js)
  2. Context enrichment       pull contact history + profile  (code_nodes/build_context.js)
  3. AI generation            Claude writes the SMS           (code_nodes/claude_generate.js)
  4. Compliance gate          TCPA / HIPAA / opt-out / freq   (code_nodes/compliance_gate.js)
  5. Delivery                 Twilio sends; log the SID
  6. Response handling        classify inbound, route         (prompts/intent_classifier.md)
  7. Outcome logging          structured record → data moat   (Supabase)
```

The structured outcome log (which objection, which strategy, what was recovered) feeds a
**vertical-intelligence** table that ranks what actually converts — so the system gets better
at each vertical over time.

## Stack

| Layer | Choice | Why |
|---|---|---|
| Orchestration | **n8n** (self-hosted, Docker) | Owns the workflow graph; self-hosting protects gross margin vs. per-op SaaS. |
| AI engine | **Claude** (Anthropic Messages API) | Per-message generation with **structured outputs** so the model can only return the SMS + metadata — no leaked preamble or reasoning. Model: `claude-sonnet-4-6` (current Sonnet — the right cost/latency tier for high-volume messaging). |
| Database | **Supabase** (Postgres + Realtime + RLS) | 11-table schema, row-level security per client, live portal updates. |
| Messaging | **Twilio** (SMS, 10DLC/A2P registered) | Compliant carrier delivery. |
| Payments | **Stripe** (+ Connect) | Inline payment links, multi-tenant. |
| Ingress / TLS | **Caddy** | Automatic HTTPS in front of n8n. |

## Compliance is in code, not in the prompt

The compliance gate (`code_nodes/compliance_gate.js`) runs **before any message sends** and is
deterministic — it does not trust the model to police itself:

- **Opt-out** — an entry in `do_not_contact` is a hard stop, honored instantly.
- **TCPA quiet hours** — no SMS outside the contact's allowed time-of-day window (timezone-aware).
- **Frequency cap** — per-contact daily message limit.
- **HIPAA** — for healthcare verticals, a PHI content filter blocks anything sensitive from
  going over SMS and routes it to a human.

Every send (and every block) is logged with its compliance flags — an auditable trail.

## Run it

### Locally, in ~10 minutes (demo, no domain)

```bash
cp .env.example .env          # fill in ANTHROPIC_API_KEY + SUPABASE_* (free tier is fine)
docker compose up -d n8n-db n8n
# open http://localhost:5678  → import the workflows from ./workflows
```

1. Create a free Supabase project. In the SQL editor, run `db/01_schema.sql` then
   `db/02_intelligence_functions.sql`.
2. Put your `SUPABASE_URL` + service-role key and `ANTHROPIC_API_KEY` in `.env`.
3. In n8n, import `workflows/trigger_ingestion.json` and `workflows/ai_response_dead_quote.json`,
   and paste the matching files from `code_nodes/` into each Code node.
4. POST a simulated missed-call / dead-quote trigger to the webhook and watch a real,
   compliant SMS get generated (and, with Twilio creds, sent).

```bash
curl -X POST http://localhost:5678/webhook/ironforge/trigger \
  -H 'content-type: application/json' \
  -d '{
        "client_id": "<a clients.id from your DB>",
        "trigger_type": "dead_quote",
        "contact": { "phone": "+15555550123", "name": "Sam" },
        "trigger_data": { "quote_amount": 8400, "scope": "full roof replacement", "quoted_on": "2026-05-30" }
      }'
```

### In production

Point a DNS A record at your host, set `N8N_DOMAIN`, and `docker compose up -d`. Caddy
provisions TLS automatically and proxies to n8n. See `ARCHITECTURE.md`.

## Status & honest scope

Iron Forge is a self-built system that was **deployed and handled real conversations in early
pilot use** for partner businesses. This repository is a clean reference rebuild of that stack
for self-hosting and review; it is **pre-revenue** (no paying customers yet), and the figures
in the schema (revenue recovered, conversion rate) are computed fields, not claims. Secrets are
templated in `.env.example` with `[REDACTED]` placeholders — nothing live is committed.

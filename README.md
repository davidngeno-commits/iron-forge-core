# Iron Forge

I built Iron Forge to recover revenue for service businesses: missed calls, dead quotes,
unpaid invoices. It's deployed, it works with real customers, and compliance is built into
the code, not bolted on. This repo is the architecture and workflows that ran in pilot.

When a business misses a call or a quote goes cold, the system texts the customer back, runs
a real conversation generated by Claude, and books the job or takes the payment. Same pipeline
for every vertical. Only three things change between a roofer and a dental office: the trigger
source, the system prompt, and the objection list.

## Pipeline

Every trigger runs the same seven stages:

1. Ingest the trigger webhook and validate it (`code_nodes/validate_payload.js`)
2. Pull contact history and client profile (`code_nodes/build_context.js`)
3. Claude writes the SMS (`code_nodes/claude_generate.js`)
4. Compliance gate: TCPA hours, opt-out, frequency, HIPAA (`code_nodes/compliance_gate.js`)
5. Twilio sends, log the message SID
6. Classify the reply and route it (`prompts/intent_classifier.md`)
7. Log the outcome to Supabase

Outcomes (which objection, which strategy, what got recovered) go into a `vertical_intelligence`
table that tracks what actually converts, and `build_context` feeds the top strategies back into
the next message.

## Stack

- **n8n**, self-hosted on Docker. Runs the workflow graph. Self-hosting keeps per-message cost
  near zero, which is what makes percentage-of-recovered-revenue pricing work.
- **Claude** (Anthropic Messages API) for generation. Structured outputs, so the model returns
  only the SMS text plus a few metadata fields and nothing else. Model is `claude-sonnet-4-6`
  (current Sonnet, the right cost tier for high message volume). Swappable per vertical via
  `CLAUDE_MODEL`.
- **Supabase** (Postgres, Realtime, RLS). 11 tables, row-level security per client, live portal
  updates.
- **Twilio** for SMS, registered for 10DLC.
- **Stripe** for payments.
- **Caddy** for TLS in front of n8n.

## Compliance

The gate (`code_nodes/compliance_gate.js`) runs before any send and doesn't trust the model to
police itself:

- Opt-out is a hard stop. A `do_not_contact` row blocks the send.
- TCPA quiet hours. No SMS outside the contact's allowed window, timezone-aware.
- Frequency cap. Per-contact daily limit.
- HIPAA. For healthcare clients, a content filter keeps PHI out of SMS and routes it to a human.

Every send and every block logs its compliance flags.

## Run it locally

```bash
cp .env.example .env        # fill in ANTHROPIC_API_KEY + SUPABASE_*
docker compose up -d n8n-db n8n
# open http://localhost:5678, import the workflows
```

1. Create a free Supabase project. Run `db/01_schema.sql` then `db/02_intelligence_functions.sql`
   in the SQL editor.
2. Put `SUPABASE_URL`, the service-role key, and `ANTHROPIC_API_KEY` in `.env`.
3. Import `workflows/*.json` into n8n and paste the matching `code_nodes/*.js` into each Code node.
4. POST a test trigger and watch it generate (and, with Twilio creds, send) an SMS:

```bash
curl -X POST http://localhost:5678/webhook/ironforge/trigger \
  -H 'content-type: application/json' \
  -d '{ "client_id": "<a clients.id>", "trigger_type": "dead_quote",
        "contact": { "phone": "+15555550123", "name": "Sam" },
        "trigger_data": { "quote_amount": 8400, "scope": "full roof replacement" } }'
```

## Known issues / not done yet

- The workflow JSON Code nodes hold a one-line stub that points at `code_nodes/`. n8n stores node
  code inline; I keep the real logic in the `.js` files and paste it in. No automated sync, so the
  two can drift if you edit one and not the other.
- No retrieval layer. Objection strategies come from seed rows in `vertical_intelligence`, not from
  each client's own conversation history. pgvector over the message log is the obvious next step.
- The pipeline is linear, not a real agent. It can't decide to call a tool mid-conversation; routing
  is fixed in the workflow graph.
- No eval harness. Prompt changes are tested by hand. No labeled set or LLM-as-judge in CI, so I can
  measure deliverability and conversion in aggregate but can't score a single prompt change.
- The HIPAA PHI filter is a regex pass. Deliberately conservative (false positives route to a human),
  but it is not a real PHI classifier.
- The example workflow assumes one Twilio number per client. Multi-number routing fits the schema but
  isn't wired into the demo graph.

## Layout

```
db/            schema + functions (RLS, engagement score, nightly rollup, onboarding, realtime)
code_nodes/    validate_payload, build_context, claude_generate, compliance_gate, engagement_scorer
prompts/       dead_quote_resurrector, missed_call_recovery, intent_classifier
workflows/     trigger_ingestion.json, ai_response_dead_quote.json (importable n8n)
docker-compose.yml, Caddyfile, .env.example
ARCHITECTURE.md, CLAUDE.md
```

## Status

Built and operated through a pilot for partner businesses. Handled real conversations and recovery
attempts. Pre-revenue, no paying customers yet. The original hosting is down, so this repo is the
clean rebuild, not a running instance. Secrets are templated in `.env.example` with placeholders.
